動手實作：從零架設 OpenClaw 開源 Agent 環境

2026-06-24

想要在自己的機器或公司伺服器上跑一個能自主執行任務的 AI Agent，但又不想把資料送到外部 SaaS？開源 Agent 框架正是答案。本文以「OpenClaw」這個開源 Agent harness 為範例，帶你從零走過一次完整的架設流程。請注意：以下指令與設定皆為「示意性質」，用來說明大多數開源 Agent 框架共通的設定模式，實際的套件名稱、參數與版本請以你所選工具的官方文件為準。

一、前置需求

在動手之前，先把基礎環境準備好。多數開源 Agent harness 的需求大同小異：

  • 作業系統：Linux（Ubuntu 22.04 以上最常見）或 macOS 皆可。若要部署到正式環境，建議使用 Linux 伺服器。
  • 執行環境：視框架而定，通常需要 Python 3.10 以上，或 Node.js 20 以上。有些框架兩者都會用到。
  • 容器：強烈建議安裝 Docker（或 Podman），後續沙箱隔離會用到。
  • Git：用來取得原始碼。
  • 模型存取：這是關鍵。你可以選擇「自架開源模型」（例如用 Ollama 或 vLLM 在本地跑），或「使用雲端 API 金鑰」（例如向某家模型供應商申請）。先決定好這一點，後面設定模型後端時會用到。

二、安裝與初始化

取得原始碼、安裝相依套件、建立設定檔，是每個框架共通的三步驟。以下為示意流程：

# 範例示意，實際指令依官方文件為準
git clone https://github.com/example/openclaw.git
cd openclaw

# 建立隔離的執行環境（Python 範例）
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

# 初始化設定檔
openclaw init

init 這類指令通常會在專案目錄產生一個設定檔（例如 config.yaml 或 .env）。建議用 venv、conda 或容器隔離相依套件，避免污染系統環境，也方便日後重現。

三、設定模型後端

Agent 的「大腦」是背後的語言模型。在設定檔裡，你需要指定模型端點與金鑰。一個典型的設定區塊看起來像這樣：

# config.yaml（示意）
model:
  provider: "openai_compatible"   # 多數框架支援相容介面
  endpoint: "https://api.example.com/v1"
  api_key: "${MODEL_API_KEY}"     # 從環境變數讀取，勿寫死
  model_name: "your-model-name"

本地 vs 雲端的取捨：若用本地自架模型（例如把 endpoint 指向 http://localhost:11434 之類的本地服務），優點是資料完全不出機房、無 API 費用，缺點是需要 GPU 與維運成本，且模型能力可能不及頂尖雲端模型。若用雲端 API，能立即取得最強模型、免維運，但要承擔費用與資料外傳的合規考量。金鑰務必透過環境變數或祕密管理工具注入，切勿提交進版本庫。

四、串接工具與 MCP

Agent 真正的價值來自「能動手做事」——讀寫檔案、呼叫 API、執行指令。框架通常透過工具（tools）或 MCP（Model Context Protocol）伺服器來提供這些能力。設定時的重點是「明確定義可用工具」與「權限白名單」：

# config.yaml（示意）
tools:
  - name: "file_read"
    allowed_paths: ["/workspace/data"]   # 只開放特定目錄
  - name: "web_fetch"
    allowed_domains: ["docs.internal.com"]  # 網路存取白名單

mcp_servers:
  - name: "internal_db"
    command: "node mcp-db-server.js"

原則是「最小權限」：只開放任務必要的檔案路徑與網域。例如一個整理報表的 Agent，沒有理由讓它能存取整個檔案系統或任意外連網路。把可存取範圍寫死在白名單裡，是控制風險最直接的手段。

五、沙箱與安全邊界

即使設了工具白名單，仍建議讓整個 Agent 在受限環境中執行。最常見的做法是用容器把它包起來：

# 範例：在容器中執行，限制資源與網路
docker run --rm \
  --memory="2g" --cpus="1.0" \
  --network none \                     # 預設不給網路，需要時再開
  -v $(pwd)/workspace:/workspace:rw \  # 只掛載工作目錄
  openclaw:latest run --task-file task.md

透過容器，你可以限制記憶體與 CPU、預設關閉網路、只掛載必要的目錄。這樣即使 Agent 因為提示詞注入或模型誤判而「失控」，破壞範圍也被限制在沙箱之內。對處理敏感資料的場景，這一層隔離不是可選項，而是必需品。

六、第一次執行一個任務

環境就緒後，就可以下第一個指令了。多數框架的執行入口長得像這樣：

openclaw run --task "讀取 /workspace/data 下的 CSV，整理出每月銷售總額，輸出成 summary.md"

送出後，觀察 Agent 的執行迴圈（agent loop）：它會先理解任務、規劃步驟、呼叫工具、檢視結果，再決定下一步，直到完成或卡住。這時請打開終端機輸出或 log 檔，逐步閱讀它的 trace——每一次的「思考、行動、觀察」。第一次跑時，建議用簡單、低風險的任務測試，確認工具呼叫、權限與輸出都符合預期，再逐步交付更複雜的工作。

七、可觀測性與下一步

能跑通只是起點。要把 Agent 用在實際業務上，可觀測性與評測不可或缺：

  • 記錄（Logging）：完整保存每次任務的輸入、推理過程、工具呼叫與輸出。這既是除錯依據，也是日後稽核與責任歸屬的基礎。
  • 評測（Evaluation）：建立一組代表性任務作為「測試集」，每次調整提示詞或更換模型後跑一遍，量化成功率與品質，避免改了一處卻悄悄退步。
  • 接到內部系統：當你對穩定度有信心後，再透過 API、訊息佇列或排程，把 Agent 接上公司的資料庫、工單系統或通訊軟體，讓它真正融入工作流。

從零架設一個開源 Agent 環境，本質上就是這七個步驟：備妥環境、安裝初始化、設定模型、串接工具、建立沙箱、執行任務、加上可觀測性。不論你最後選用的是 OpenClaw 還是其他框架，這套模式都通用。建議先在本地小規模跑通，把安全邊界與權限白名單設好，再逐步擴大應用範圍——穩紮穩打，才能放心把 Agent 交付到真實業務之中。
