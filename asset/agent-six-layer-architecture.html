Agent 六層架構：從模型到治理

2026-05-02

了解了 Chat（聊天機器人）與 Agent（能自己動手完成任務的 AI 代理人）的差別之後，下一個老闆最常問的問題是：「那我要怎麼把一套 Agent 系統真正蓋起來，而且不會三個月後就壞掉、沒人敢動？」很多企業興沖沖地做了一個 Agent，結果發現它難維護、出了錯不知道是哪裡的問題、想擴大規模又得整個打掉重練。

問題的根源，幾乎都是「沒有分層」。就像蓋房子，不會把水電、結構、裝潢全攪在一起施工；經營公司，也不會讓一個人同時管採購、生產、財務、法務。一套穩定可靠的 Agent，背後是一個有秩序的「六層架構」——從最底下的「大腦」一路往上疊到最頂端的「管理規範」。本文用蓋房子和公司組織分工的比喻，帶你把這六層一次看懂，也看懂為什麼少了哪一層都會出事。

分層的最大價值：出事時你知道是哪一層的問題，而不是整棟一起塌。下面逐層拆解。

六層架構全景圖：像一棟蓋好的房子

請把 Agent 系統想像成一棟六層樓的房子，由下往上蓋：地基要先穩，才能往上加樓層，最後屋頂裝上監視器與門禁。每一層都有清楚的職責，也都有「如果這層偷工減料會發生什麼事」。

01

模型層 Model Layer — 大腦

選一顆「會思考的腦袋」，也就是基礎大型語言模型（會讀懂與生成文字的 AI 引擎，例如 Claude、GPT、Llama）。

02

推理層 Reasoning Layer — 判斷力

把大腦的想法變成一步步的決定：先做什麼、再做什麼、卡住了怎麼辦。

03

工具層 Tools Layer — 手腳

讓 AI 真的能動手：查資料庫、寄信、開發票、呼叫你公司的系統。

04

狀態層 State Layer — 記憶

記得剛剛做到哪、客戶說過什麼，不必每次都從零開始重講一遍。

05

執行層 Execution Layer — 工頭

協調多個步驟有序進行：誰先誰後、超時了怎麼辦、失敗了要不要重試。

06

治理層 Governance Layer — 管理規範

監控、紀錄、權限、審批——確保 AI 不會闖禍，出事也查得到原因。

一句話記住

下面三層（大腦、判斷力、手腳）讓 Agent 「能做事」；上面三層（記憶、工頭、管理規範）讓 Agent 「做得可靠、做得久、可以信任」。很多公司只蓋了下面三層就上線，這正是後來「無法維護、治理混亂」的原因。

各層詳解：對老闆而言真正該知道的決策點

第 1 層・模型層：先租大腦，還是自己養大腦？

這層的核心抉擇，是要用「API 模型」（向供應商按使用量付費、隨開隨用，例如 Claude、GPT）還是「開源模型」（把模型放在自己機器上自己維護，例如 Llama、Mistral）。用租房子來比喻：API 模型像租屋，搬進去就能住、不必操心修繕，但每月要付租金；開源模型像買房，長期住下來成本可控、完全自己掌握，但水電維修都得自己扛。

實務建議：初期用 API 模型快速驗證點子，因為它開發快、能力強，最適合「先看到成效」。等到用量真的很大（例如每月超過百萬字級的處理量），再評估搬到開源模型或混合方案來壓低長期成本。一開始就執著自建，往往是還沒驗證價值就先燒掉大筆預算。

第 2 層・推理層：這就是 Agent 和「會聊天的 AI」最大的差別

聊天機器人只會回答你問的；推理層讓 AI 會自己「拆任務、做決定、遇到問題換條路走」。它的四個關鍵能力是：把複雜任務拆成小步驟、決定何時該用哪個工具、依中間結果決定下一步、以及工具失敗時自動找替代方案。常見做法叫 CoT（思考鏈，要求 AI 把推理過程一步步寫出來，比較不會亂跳結論）和 ReAct（一邊想一邊動手的框架）。代價是每做一次決策，AI 要多花一點「思考成本」（大約相當於多讀寫幾百字的費用）。

第 3 層・工具層：沒有工具，AI 只會出一張嘴

工具層是 Agent「改變世界的手腳」。沒有工具，再聰明的 AI 也只能跟你聊天，不能真的幫你開發票、回客戶、更新庫存。常見工具包含：呼叫第三方服務的 API（像串接你的 ERP、物流系統、金流）、查改企業資料庫、執行程式、讀寫檔案、寄送郵件與通知。關鍵心法是：每個工具都要有清楚的「使用說明書」（這個工具做什麼、要填什麼參數、會回傳什麼），AI 才不會拿錯工具或填錯資料。一個涵蓋典型企業日常的 Agent，常需要數十個工具才夠用。

第 4 層・狀態層：別讓你的 AI 是金魚腦

沒有記憶的 AI，每次都要你把背景重講一遍——客戶剛報過的需求、訂單做到哪一步，全忘光。狀態層就是 Agent 的工作筆記本，記住：任務目標與進度、已蒐集的資訊、中間決策、失敗重試了幾次、以及客戶的偏好與歷史。技術上通常用「記憶緩衝」處理短期、用「向量資料庫」（能依語意快速找回相關內容的資料庫）處理長期記憶。這層會讓儲存與查詢成本增加約兩、三成，但換來的是順暢、不重複的體驗，對客服與業務場景尤其值得。

第 5 層・執行層：工地需要一個工頭

當一個任務要動用好幾個工具、好幾個步驟，就需要「工頭」來指揮：決定步驟先後（哪些可以同時做、哪些要等前面完成）、設定超時（避免某一步卡住整條流程）、設定失敗自動重試、以及「優雅降級」（核心工具掛了就啟用備案，而不是整個倒下）。最佳實踐是直接採用成熟的工作流引擎（如 Temporal、Airflow），而不是自己土法煉鋼，省下大量穩定性的踩坑。

第 6 層・治理層：上線前最容易被省略、卻最不該省的一層

當 Agent 開始碰公司的錢、客戶資料與對外決策，「失控」的代價就不再是小事。治理層有四根支柱：權限與審批（哪些決定需要人簽核、誰能批）、監控與告警（異常行為即時被抓到）、日誌與審計（每個動作都留紀錄、可回溯究責）、可解釋性（業務同仁看得懂 AI 為什麼這樣決定）。這一層就像公司的稽核與法務——平常感覺不到它的存在，出事時你會慶幸當初有裝。

最常見的致命省略

九成「Agent 翻車」事件，都是因為只蓋了第 1–5 層、跳過第 6 層治理就急著上線。等到 AI 寄錯信、改錯帳、洩漏資料，才發現沒有日誌可查、沒有權限可擋、沒有審批可攔。治理不是上線後的選配，是上線的前提。

為什麼「分層」這麼重要？

分層不是工程師的潔癖，而是直接關係到你的維護成本、風險與擴張速度。用一個對照就很清楚：把六層攪成一團的系統，就像一間沒有部門、沒有組織圖的公司，出了事沒人知道該找誰；分好層的系統，出事時你能立刻指向某一個「樓層」去修。

實施順序：別想一次蓋完六層

蓋房子不會六層同時動工，導入 Agent 也一樣。正確做法是「由下往上、分批驗證」，每個階段都先確認站得住，再往上加。下面是一個約六週、適合中小企業試點的節奏：

常見誤區

小心這三個坑

① 跳過治理層就上線——前面五層都做了，卻沒做日誌、權限與審批，等於開快車不裝煞車。
② 一開始就想自建模型——還沒驗證價值就投入大筆預算養大腦，常常燒錢又拖慢進度。
③ 六層同時開工——範圍越大、變數越多，越容易卡住又找不到問題出在哪一層。

給老闆的一句話

重點

不要問「我們有沒有做 Agent」，要問「我們的 Agent 蓋到第幾層了」。能做事的下三層只是入場券，能信任的上三層才是能不能放進公司日常營運的關鍵。從一個最痛、最能量化的流程開始，由下往上分批蓋、每層都驗證過再往上——這才是讓 AI 真正落地、而且不會半年後崩掉的方法。

結語

六層架構模型，就是建立一套可靠、可治理 Agent 系統的藍圖：從選對大腦開始，補上判斷力與手腳讓它能做事，再加上記憶、工頭與治理讓它做得久、值得信任。先求站得穩，再求蓋得高。下一篇我們會深入第 3 層工具層，介紹 Agent 的核心零件——MCP、Skill、Sandbox、Plugin，帶你看看這些手腳實際上是怎麼組裝起來的。
