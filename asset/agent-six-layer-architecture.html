Agent 六層架構：從模型到治理

2026-05-02

了解了 Chat 與 Agent 的區別後，下一個問題是：Agent 系統該怎麼構建？許多企業建立 Agent 後發現難以維護、無法擴展、治理混亂。根據 Asgard AI 與業界最佳實踐，成功的 Agent 系統遵循「六層架構模型」，從底層的模型選擇到頂層的治理與重用，環環相扣。本文詳解這個模型。

六層架構全景圖

Agent 系統並非簡單堆砌元件，而是有序的分層結構：

底層 → 頂層的六層：
1️⃣ 模型層（Model Layer） — 選擇基礎大型語言模型
2️⃣ 推理層（Reasoning Layer） — 決策邏輯與思考過程
3️⃣ 工具層（Tools Layer） — API、資料庫、系統集成
4️⃣ 狀態層（State Layer） — 記憶與上下文管理
5️⃣ 執行層（Execution Layer） — 工作流程與行動協調
6️⃣ 治理層（Governance Layer） — 監控、日誌、權限、可審計性

各層詳解與設計決策

第 1 層：模型層（Model Layer）

核心決策：選擇開源模型（Llama、Mistral）還是 API 模型（Claude、GPT-4）？

• 開源模型 — 完全控制、成本可預測、適合長期規模化但需自行維護
• API 模型 — 開發快速、能力強、適合快速試點但需按次付費且依賴供應商

建議：初期用 Claude API 快速驗證，成功後如量級大（>100 萬 tokens/月），考慮遷移至開源模型或混合方案。

第 2 層：推理層（Reasoning Layer）

核心功能：將模型的回應轉化為結構化決策。這是 Agent 與 Chat 最大的區別。

關鍵能力：
• 目標分解 — 複雜任務分解為子步驟
• 工具選擇 — 決定何時呼叫哪個 API
• 條件判斷 — 根據中間結果決定下一步
• 錯誤恢復 — 若工具調用失敗，自動尋找替代方案

實現方式：Chain of Thought（COT）、ReAct 框架等。成本：每次決策 100-500 tokens 的思考開銷。

第 3 層：工具層（Tools Layer）

本質：Agent 用來改變世界的手臂。沒有工具，Agent 只能聊天。

工具類型：
• REST API — 調用第三方服務（Slack、Salesforce、Stripe）
• 資料庫查詢 — 查閱或修改企業資料
• 程式碼執行 — 運行自訂邏輯（Python、SQL）
• 檔案操作 — 讀寫檔案系統
• 通知系統 — 寄送郵件、訊息、報告

關鍵：每個工具必須有清晰的文檔（描述、參數、返回值），Agent 才能正確使用。通常需要 50-100 工具才能覆蓋典型企業場景。

第 4 層：狀態層（State Layer）

問題：Agent 無法記住上下文，同樣的信息需要一遍遍輸入，效率低下。

解決方案：建立狀態管理機制，記錄：
• 任務目標與進度
• 已蒐集的資訊
• 中間決策結果
• 失敗重試次數
• 用戶偏好與歷史互動

實現方式：Memory buffer（短期）+ 向量資料庫（長期）。成本：每個工作流增加 20-30% 的儲存與查詢成本。

第 5 層：執行層（Execution Layer）

功能：協調多個工具與步驟，確保工作流有序執行。

核心設計：
• 步驟調度 — 決定執行順序，允許並列或條件分支
• 超時控制 — 防止單一步驟無限等待
• 重試邏輯 — 失敗自動重試，次數可配
• 優雅降級 — 若核心工具失敗，啟用備選方案

最佳實踐：使用工作流引擎（Airflow、Temporal）而非自行編寫，保證穩定性。

第 6 層：治理層（Governance Layer）

為何需要治理？Agent 在企業中掌控敏感決策與資料，失控可能造成重大損失。

治理四支柱：
• 權限與審批 — 哪些決策需要人工審批？誰有權批准？
• 監控與告警 — 異常決策即時發現
• 日誌與審計 — 完整記錄所有決策與行動，可回溯
• 可解釋性 — Agent 決策的原因可被業務人員理解

層層遞進的實施順序

不推薦一下子建完所有六層。正確的順序是：

第 1-2 週：第 1-3 層
Choose model + 簡單決策邏輯 + 接入 2-3 個關鍵工具。目標：驗證可行性。

第 3-4 週：加入第 4 層
加入狀態管理，讓 Agent 能記住上下文。此時工作流變得更自然。

第 5-6 週：完善第 5-6 層
搭建執行框架與治理系統。此時 Agent 可投入生產。

投資與成本估算（台灣中型企業）：
• 第 1-3 層：$200 萬（技術驗證）
• 第 4-5 層：$300 萬（核心系統）
• 第 6 層：$200 萬（治理與安全）
• 總計：$700 萬，投資回收期 8-12 個月

六層架構的優勢

1. 清晰的分工 — 各層職責明確，不同角色各司其職
2. 漸進式投資 — 可按階段推進，風險逐步降低
3. 可重用性 — 下層組件可被多個 Agent 共用
4. 可維護性 — 問題易定位，修改隔離性好
5. 可擴展性 — 新增工具或規則不需重構整個系統

結語

六層架構模型是建立可靠、可治理 Agent 系統的藍圖。從模型選擇開始，層層遞進，最終達到企業級的監控與治理。下一篇將深入第 3 層，介紹 Agent 的核心元件：MCP、Skill、Sandbox、Plugin，說明如何實際組裝這些零件。
