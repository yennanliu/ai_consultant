Agent 核心元件與實施路線

2026-05-02

前兩篇我們談了 Agent 是什麼（會自己思考、自己動手完成任務的 AI，而不只是會聊天的機器人），也畫了它的架構藍圖。這一篇要回答最實際的問題：真正要動手蓋一個 Agent，到底是用哪些零件組起來的？

答案是四個核心元件：MCP、Skill、Sandbox、Plugin。名字聽起來很技術，但其實每一個都對應一件你公司裡早就在做的事。為了讓老闆聽得懂，我們先用一個比喻把四者一次串起來：把 Agent 想像成你新請的一位數位員工——

01

MCP　＝　員工的「門禁卡」

讓這位數位員工能用標準方式去刷進公司各個系統（CRM、ERP、出貨平台、Email），查資料、按按鈕。一張卡就能進所有門。

02

Skill　＝　員工的「工作 SOP」

一套套教好的標準作業流程。例如「做月報」這個 Skill，就是把查數字、比上月、算成長、寫結論這幾個動作，包成一個按鈕。

03

Sandbox　＝　員工的「實驗室」

一間與正式系統隔開的安全房間。要試算、跑程式、做複雜運算，都在這裡進行，搞砸了也不會碰壞公司真正的帳務或資料。

04

Plugin　＝　可隨時插拔的「外掛配件」

像手機 App 一樣，要哪個功能就裝哪個、不要就關掉，全程不用把整套系統關機重來。

下面這張對照表，把四個元件用同一組問題拆給你看——它是什麼、它幫你解決什麼麻煩、實際長什麼樣子。看懂這張表，這篇文章的一半就吸收了。

四大核心元件對照：是什麼 / 解決什麼問題 / 例子
元件白話講是什麼解決什麼問題具體例子

MCP：讓 AI 一張卡刷遍所有系統

MCP（Model Context Protocol，模型情境協議）是 Anthropic 提出的一套公開標準，規範了「AI 模型要怎麼跟外部工具講話」。你可以把它想成水電的國家標準——只要插頭符合規格，任何電器插上任何插座都能用，不必為每台電器各拉一條專線。

對中小企業最直接的意義是：以前每接一個新系統（換一家物流、導一套新的進銷存），工程師都得重寫一段「翻譯程式」，費用與時間都重來一次。有了 MCP 這個共通標準，接新系統像換插頭，不是重新拉線。

它為老闆省的，主要是這四件事：

• 標準化——不論對方是 LINE、SQL 資料庫還是你家的舊 ERP，都用同一種方式接入。
• 可互換——今天用這套 Agent 框架接好的工具，明天換框架也能繼續用，不被單一廠商綁死。
• 內建安全——權限管控、錯誤處理、操作紀錄都是規格的一部分，誰在什麼時候動了什麼，查得到。
• 更省成本——溝通格式精簡，比傳統土法接法明顯減少 AI 的運算花費（也就是每次呼叫的 token 用量），長期帳單更輕。

紡織業例子

一家布料貿易商，過去業務查庫存得開 ERP、查客戶得開 CRM、確認船期得登入報關平台，三個系統三套登入。導入 MCP 後，業務只要在對話框問「A 客戶這款卡其布還有多少現貨、最快哪天能出」，Agent 用同一張「門禁卡」一次刷過三個系統，幾秒鐘給出整合答案。

什麼時候該用 MCP？只要任務需要「去查公司既有系統的資料」或「在外部系統按下某個動作」，就一定會用到它。常見如：接 CRM 查客戶往來紀錄、接通訊軟體發通知、接內部知識庫翻文件規章。
大概要花多少？每接一個系統大約是 4–8 小時的工，視系統複雜度，單一接點費用通常落在數萬元等級——而且接得越多，攤下來的單價越低。

Skill：把「整套流程」打包成一個按鈕

很多人會把 Skill 和「程式功能」搞混。差別在這裡：單一功能只做一件事，Skill 是把一串為了完成某個業務目標的動作，包成一個可重複使用的整套流程。

用「做月報」舉例最清楚：

單一功能 vs Skill：差在哪？
比較單一功能（一個動作）Skill（一整套 SOP）
做的事「查本月銷售額」回傳一個數字查本月 → 比上月 → 算成長率 → 標出異常 → 寫成一頁結論
誰來串起來每次都要人重新交代步驟步驟已內建，一句「做月報」就跑完
能不能重用只是個零件✓ 不同部門、不同 Agent 都能直接套用
一句話總結：功能是「零件」，Skill 是把零件組好、可以直接上工的「整套工法」。

一個設計良好的 Skill 通常有三層：輸入（要給它什麼，例如月份、產品線）、處理流程（依序查資料、運算、判斷）、輸出（給出報表、表格或一段白話結論）。建構時把握三個原則：一個 Skill 對應一個明確的業務目標（不要塞太多）、把它寫成可版本控管的設定檔（改了什麼有紀錄）、並逐步累積成一座「Skill 庫」。

為什麼 Skill 最划算

開發一個 Skill 大約要 1–2 週，但它一次做好、能被多個部門和多個 Agent 重複使用。同一個「客訴分類」Skill，客服、業務、品保都能用——做一次、回收很多次，這是四大元件裡投資報酬最明顯的一個。

Sandbox：給 AI 一間「弄壞了也沒關係」的實驗室

Agent 厲害的地方，是它有時會自己寫一小段程式來算複雜的東西（例如把上萬筆訂單做交叉統計、跑一份財務試算）。問題來了：讓 AI 直接在公司正式系統上跑沒看過的程式，風險太高——萬一邏輯出錯、無限佔用資源，或不小心碰到敏感資料，後果不堪設想。

Sandbox（沙箱）就是為此而生的隔離房間。AI 想跑的程式全部關進這間實驗室執行，跑壞了、跑爆了，都被關在房間裡，碰不到外面真正的帳務與客戶資料。

一間設計妥當的 Sandbox 會設下四道防線：資源上限（限制能用多少運算力與時間，避免一支程式拖垮系統）、檔案隔離（看不到也改不了系統機密檔）、網路管制（只准連事先核可的網址）、逾時中止（跑太久立刻喊停）。技術上常用 Docker 容器（最普遍、成本低）、虛擬機（隔離更徹底但較吃資源），或雲端函式（不必自己養機房）。

會計事務所的提醒

一家會計事務所用 Agent 自動跑客戶試算表，若沒有 Sandbox，一段算錯的程式可能直接覆蓋客戶的真實底稿。有了 Sandbox，所有試算都在副本上進行，確認無誤才回寫。凡是會碰到錢、碰到客戶機密資料的流程，Sandbox 不是加分項，是必備品。

Plugin：像裝手機 App 一樣，隨插隨拔

Plugin（外掛）讓你的 Agent 系統能在不關機、不重新部署的情況下，臨時加上或拿掉某個功能。和 Skill 的差別是：Skill 比較像「員工已經學會的能力」，部署當下就定好；Plugin 則像「可隨時插拔的配件」，營運中就能開關。

對老闆來說，Plugin 真正的價值是讓「試新功能」變得低風險：

• 小範圍試水溫——新報表先給三家門市用，好用再全面開。
• 分批上線——新功能先推 10% 客戶，沒問題再擴大，出事影響面也小。
• 客製不打架——不同客戶裝不同 Plugin，互不干擾，也不必為一家客戶改動整套系統。

四個元件怎麼串起來幹活？

單看四個零件可能還是抽象。我們用一個真實的請求，看它們如何接力。情境：餐飲連鎖的老闆在對話框打了一句「幫我分析上一季各分店的營收表現，做成一頁報告寄給我」。

01

老闆下指令

用自然語言提出任務，不需要懂任何技術細節。

02

Agent 選對 Skill

判斷這是「季度營收分析」這個 Skill 的工作，調出對應的 SOP。

03

用 MCP 撈資料

透過 MCP 這張門禁卡，刷進 POS 與會計系統，取出各分店的營收數字。

04

進 Sandbox 運算

在隔離實驗室裡跑統計與圖表，比較分店、算出成長率、標出異常。

05

靠 Plugin 交付

用 Email Plugin 把報告寄出，同時推上 Dashboard Plugin 供隨時查看。

環環相扣：MCP 負責進得去、Skill 負責知道怎麼做、Sandbox 負責安全地算、Plugin 負責把成果送出去。少了任何一環，這位數位員工都做不完一件完整的事。

六個月落地路線圖

聽起來很完整，但中小企業最怕的就是「一次想做完、結果做不完」。我們建議的節奏是分四階段、半年到位，每一階段都先產出看得到的成果，再往下走。下面這張圖把六個月的累積進度視覺化——重點不是哪個月花多少錢，而是價值是逐月疊上去的，不是最後一刻才出現。

每一階段都交付可用成果：先接得進系統，再學會做事，再安全運算，最後實際上線驗證。
01

第 1 個月　打地基（MCP）

盤點現有系統，挑出 3–5 個最關鍵的工具（如 CRM、進銷存、通訊軟體），逐一用 MCP 接通。先求「AI 進得了你的系統」。

02

第 2–3 個月　教技能（Skill）

針對最痛、最高頻的流程，開發 5–8 個業務 Skill（例如銷售分析、客訴分類、報價生成），並建立可版本控管的 Skill 庫。

03

第 4 個月　築防線＋裝配件（Sandbox + Plugin）

搭好 Sandbox 安全環境，讓需要運算的任務有地方安全執行；再設計 2–3 個關鍵 Plugin（通知、報表、儀表板）方便交付。

04

第 5–6 個月　小規模上線驗證

先在一個部門或少數門市實際運行，蒐集真實回饋，監控成效並持續調校，確認可行再逐步擴大。

給老闆的關鍵心法

不要把六個月當成「等半年才看得到東西」。正確做法是每個階段結束都要有一個能向團隊展示的成果——第一個月就能 Demo「AI 幫我查到了庫存」，第三個月能 Demo「AI 自動做出了月報」。讓成效一段一段被看見，內部支持才會一段一段累積起來。

三個最常見的踩雷點

小心這些陷阱

① Skill 設計貪心——一個 Skill 想包山包海，結果又難維護又容易出錯。守住「一個 Skill = 一個明確業務目標」。
② Sandbox 沒設好退路——程式跑失敗或逾時，整條流程卡死。一定要先想好逾時、重試與「失敗了改用人工」的降級方案。
③ Plugin 版本打架——外掛之間互相依賴，更新一個就拖垮另一個。嚴格做版本管理與相容性測試，別讓便利變成災難。

給老闆的一句話

MCP、Skill、Sandbox、Plugin 聽起來像四個技術名詞，本質上只是回答了四個經營問題：AI 進得了我的系統嗎（MCP）？它知道怎麼把事做對嗎（Skill）？它出錯時會不會弄壞東西（Sandbox）？我能安心地一點一點開放它嗎（Plugin）？

四個答案都「是」，你就有了一位可靠的數位員工。而通往那裡的路，不是賭一把砸大錢一次到位，而是分階段、每一步都先讓成效被看見，再決定要不要往前走——這也正是我們協助每一家中小企業導入 Agent 的核心原則。
