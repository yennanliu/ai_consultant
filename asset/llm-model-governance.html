模型治理：從模型選型到生命週期的全程管控

2026-06-05

當企業開始大量導入大型語言模型（LLM，Large Language Model，就是 ChatGPT、Claude 這類能讀寫文字的 AI），模型本身就成了一項必須被管理的資產。哪些團隊用了哪些模型？版本是否過期？敏感資料流向了哪裡？哪天供應商把舊版本關掉、服務會不會突然中斷？萬一 AI 給錯答案造成損失，誰要負責？這些問題若沒有一套制度回答，AI 應用很快就會變成沒人說得清、也沒人管得動的「影子系統」（Shadow IT，員工私下使用、IT 部門卻不知情的工具）。

本文要說的模型治理（Model Governance），用一句話講就是：把每個 LLM 都當成一名「受監管的新進員工」來管理。你不會讓一位來路不明、沒人面試、沒人帶、沒人考核的員工直接接觸客戶與帳務——模型也一樣。下面我們會從盤點、選型、上線、監控到汰換，一步步說明中小企業如何用輕量的方式把這件事做好。

員工管理對應的模型治理動作
人事名冊模型清單：登錄用了哪些模型、版本、供應商、用途

什麼是模型治理

模型治理是指企業以制度化的方式，把模型當成受管控的企業資產來管理——涵蓋選型、上線、監控、汰換的整個生命週期。它和既有的治理體系緊密相關：資料治理（Data Governance）管的是資料的品質、權限與流向（哪些資料能用、誰能看）；一般 IT 治理管的是系統的安全與變更流程；而模型治理則疊在兩者之上，專門處理「模型」這個新型資產特有的問題。打個比方：資料治理像管倉庫裡的原料，IT 治理像管工廠的機台與門禁，模型治理則是管那位會自己做判斷、還會不斷「換腦袋」的新員工。

為什麼 LLM 特別難管？原因有三。第一是黑箱特性：模型的決策過程難以完全解釋，你無法像看傳統程式一樣逐行追蹤它「為什麼這樣回答」。第二是快速演進：供應商幾乎每隔幾個月就推出新版本、棄用舊版本，企業被迫不斷重新驗證——就像你才剛把員工訓練好，他的「腦袋」又被悄悄換了一顆。第三是外部相依：多數中小企業透過商用 API（應用程式介面，指付費呼叫雲端模型的服務）取用模型，模型的行為、定價、能不能用，全掌握在第三方手中。

重點

模型治理不是要你「少用 AI」，而是讓你用得更放心、更大膽。沒有治理，一次資料外洩或一筆算錯的報價，就可能賠掉好幾個月省下來的人力成本。

模型治理的生命週期：五個環節

不論企業大小，模型治理都繞著同一條主軸運轉：從挑選、把關、上線、監督，到最後的更新汰換，週而復始。先看懂這條流程，再深入各環節就容易得多。

01

選型

依任務需求比較不同模型的能力、成本、延遲與資料政策，列出候選名單。

02

准入

通過資安與資料審查、指定業務負責人、登錄進模型清單後，才正式放行。

03

上線

釘選（pin）特定版本、建立模型卡、依風險等級設定審批與人工把關。

04

監控

持續觀察品質漂移、成本與異常，保留決策日誌以利稽核與問題排查。

05

汰換

供應商棄用舊版或出現更好選擇時，排程遷移、重新驗證，回到選型。

模型清單與盤點

治理的起點是「知道自己在用什麼」。許多企業嚴重低估了這一步——當業務部串了 ChatGPT 寫文案、客服串了另一家做自動回覆、會計又私下用某個工具整理發票時，往往沒有任何一個人說得清公司到底在用哪些 AI、餵了哪些資料進去。這就像公司裡來了一群沒登記在人事名冊上的約聘人員，誰都管不到。

建立模型清單（Model Inventory，也常稱 Model Registry，模型登錄簿）就是要把這些攤在陽光下，逐一登錄。一份及格的清單至少要回答下表這幾個欄位：

模型清單該登錄哪些欄位
欄位說明範例
模型與版本用了哪個模型、釘選哪個版本Claude（指定版本號）
供應商來自哪家、透過什麼管道取用商用 API
業務用途用在哪個流程、解決什麼問題客服首次回覆草稿
資料流向餵了哪類資料、送去哪裡、是否含個資客戶詢問（去識別化）
風險等級高／中／低，決定審查強度中
業務負責人誰為這個應用的成效與風險負責客服主管
中小企業用一份試算表就能起步，重點是每一欄都有人填、有人定期更新。

這份清單應由專人維護、定期更新。它不只是文件，更是後續所有治理動作的地基：沒有清單，就無法評估整體風險、無法在供應商棄用某版本時即時反應、也無法在資料外洩事件發生時，第一時間釐清「哪些應用、哪些客戶受到波及」。

模型選型與准入

新模型進入企業前應通過准入評估，而不是由某位工程師憑個人喜好直接串接上線——這就像錄取一名新員工前要先面試，不能讓任何人把朋友直接帶進公司碰帳務。評估維度至少包括五項：能力（是否真能勝任這項任務）、成本（每百萬 token 的價格與預估用量；token 是模型計費的文字單位，大約幾個字算一個 token）、延遲（回應速度是否符合場景，客服要快、夜間批次跑報表則可以慢）、授權條款（商業使用是否合規），以及最關鍵的資料政策（供應商會不會拿你的輸入去訓練它的模型、你的資料存在哪個國家）。

一個常見的關鍵抉擇是自架開源模型 vs. 商用 API。商用 API 上手快、品質穩定、免於維運，但資料須送出企業、且受制於供應商；自架開源模型（如 Llama、Qwen 等可免費下載、自行部署的模型）能讓敏感資料完全留在內部、長期成本可控，但需要 GPU 主機與專業團隊，品質通常還得額外調校。下表幫你快速對照：

商用 API vs. 自架開源模型
比較項目商用 API自架開源模型
上手速度✓ 幾天即可串接✗ 需建置與調校
資料是否留在內部✗ 須送出企業✓ 完全自主可控

版本控管與生命週期

模型不是「一次上線、永遠不變」。每個模型都應有清楚的生命週期：上線 → 版本標記 → 棄用（deprecation，供應商宣告即將停止支援）→ 汰換。供應商通常會提前公告舊版本的停用日期，企業必須據此排程遷移，否則服務可能在某天毫無預警地中斷。

這裡有個務實又重要的小技巧：盡量釘選（pin）特定版本，而不是永遠使用「latest（最新版）」。因為供應商一旦悄悄更新，模型的行為可能改變、甚至在你的特定任務上品質回退（明明之前好好的，突然變笨）。釘版本，等於是讓你的「員工」不會一夜之間被換了腦袋還沒人通知你。

另外，每個納管的模型都應有一份模型卡（Model Card）——可以理解成這名員工的「職務說明書＋體檢報告」，記錄它的用途、能力邊界、已知限制、評測結果與適用／禁用場景。模型卡讓接手的人不必從頭摸索，也是日後稽核時的重要依據。

風險分級與審批

不是所有 AI 用途都需要相同強度的審查。把治理資源平均灑在每個應用上，只會讓繁瑣流程拖垮所有 AI 嘗試。正確做法是依用途的風險高低，決定審查強度——低風險的快速放行，高風險的嚴格把關。

用途風險分級：審查強度與範例
風險等級審查強度典型範例
高風險須人工審核＋正式簽核，上線後維持人在迴路監督財務決策建議、診所衛教或用藥資訊、直接發給客戶的報價／合約條款
中風險需登錄與抽查，輸出由員工確認後才使用客服回覆草稿、行銷文案、合約初步摘要
低風險簡化流程、快速放行內部會議記錄摘要、個人草稿潤飾、腦力激盪
「人在迴路（human-in-the-loop）」指 AI 出結果後，由真人複核才放行——高風險場景的最後一道防線。

舉例來說：一家會計事務所用 AI 摘要內部會議記錄，是低風險，員工自行使用即可；但若用 AI 直接產生報稅建議發給客戶，就是高風險，必須由會計師逐筆複核並簽核。一家診所用 AI 整理內部排班，無妨；但用 AI 回答病患的用藥問題，就必須有醫護人員把關。分級的精神，是讓低風險創新跑得快，把人力集中在真正可能造成傷害的地方。

監控、稽核與責任歸屬

模型上線不是終點。最容易被忽略、卻最危險的，是漂移（drift）——模型的輸出品質會隨時間或資料變化而悄悄退化。打個比方：一名員工剛到職時表現很好，但客戶的問法變了、市場的行話變了，他卻沒跟著更新，答得就漸漸不對勁了。模型也一樣，需要定期「回頭考核」。

上線後的監控重點有三：品質監控（回應準確度、客訴率是否上升）、成本監控（token 用量與費用是否異常飆升，避免月底收到嚇人的帳單），以及決策日誌（保存關鍵決策的輸入、模型版本、輸出與後續結果，方便日後排查與稽核）。

比技術監控更根本的問題是責任歸屬：當模型給出錯誤建議造成損失，誰負責？治理制度必須為每個 AI 應用明確指派一位業務負責人（owner），由他為該應用的成效、合規與風險負責，而不是把責任模糊地推給「AI」或 IT 部門。沒有負責人的 AI 應用，就像沒有主管的員工——出了事沒人收尾。

政策與組織

治理要真正落地，需要白紙黑字的使用政策，明確規範哪些資料可以丟給外部模型、哪些絕對不行（例如客戶身分證號、未公開財報、營業祕密）、哪些用途需要事前申請。同時建議成立一個跨職能治理小組，成員涵蓋技術、業務、法務與資安，定期審視模型清單、審批高風險用途、檢討事件。

關鍵原則是：模型治理不能另起爐灶，要與既有的合規與資安流程銜接——例如把模型准入直接掛進原本就有的供應商審查與資安評估表單裡，而不是再造一套沒人想用的新流程。

小心

最常見的三個翻車情境：① 影子 AI 失控——各部門私接模型，敏感資料早已外流卻無人知曉。② 永遠用 latest——供應商更新後行為改變，品質默默回退也沒人發現。③ 沒人負責——出事時才發現這個應用根本沒有 owner，無人收尾。

給中小企業的輕量起步建議

中小企業沒有大型企業的治理人力，但這不代表可以略過治理；關鍵是輕量、夠用、漸進。你不需要一開始就買昂貴的治理平台，下面這個四步順序，多數公司一個月內就能跑起來：

01

建清單

用一份試算表，登錄目前在用的所有模型、版本與業務用途，指派每項的負責人。

02

寫政策

寫一頁就好的使用政策，劃出資料紅線：哪些資料絕對不能丟給外部模型。

03

分級把關

挑出少數高風險用途，加上人工審核與簽核流程，其餘維持輕便。

04

補監控

對重點應用定期抽查品質與成本，再逐步補上模型卡與決策日誌。

最大的風險從來不是「治理做得不夠完美」，而是「完全沒有治理、不知道自己在用什麼」。先把基礎建起來，後面再慢慢補強。

給老闆的一句話

模型治理的本質，是讓企業在享受 LLM 帶來的生產力之餘，仍然清楚知道：自己在用什麼、承擔什麼風險、由誰負責。把每個模型都當成一名受監管的員工——有名冊、有面試、有職務說明、有考核、有主管——你就不會在某天被一筆算錯的報價或一次資料外洩打個措手不及。

建議

不必等到「準備好」才開始。今天就開一份試算表、把現在公司在用的 AI 工具列出來，再寫下三條資料紅線——這就是你的模型治理 1.0。從一張清單和一頁政策起步，永遠勝過等到出事才補救。
