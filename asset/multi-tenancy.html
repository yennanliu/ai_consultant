多租戶 LLM 系統設計：隔離、配額與成本分攤

2026-06-12

當一個 AI 平台需要同時服務多個客戶、多個部門，或多個品牌時，「多租戶（multi-tenancy，一套系統同時服務多組獨立使用者）」就成為架構的核心議題。想像你蓋了一棟辦公大樓：水電、電梯、保全都是大家共用的，但每一間公司有自己上鎖的門、自己的鑰匙、自己的水電帳單——這就是多租戶。相對地，每家客戶都各蓋一棟「獨棟透天」，安全感最高，但蓋十棟的成本也是十倍。

對中小企業與 SaaS 團隊（軟體即服務，按月訂閱使用的軟體生意）而言，能不能把一套 LLM（大型語言模型，像 ChatGPT 背後那種會讀會寫的 AI）系統設計成「可安全共用、算得清楚、控得住成本」的平台，往往決定了你能服務 10 個客戶、還是 1,000 個客戶。本文用白話拆解多租戶 LLM 系統的五個關鍵設計，以及最容易踩的雷。

什麼是多租戶，為什麼 LLM 平台常需要它

多租戶指的是同一套軟體與基礎設施，同時服務多個彼此獨立的「租戶（tenant，可以想成大樓裡的一家公司）」——可以是不同的付費客戶，也可以是企業內不同的部門或事業群。每個租戶在登入後看到的，都是「自己專屬的系統」，但底層的運算、AI 模型呼叫與資料儲存其實是大家共用的。

LLM 平台特別需要多租戶，原因在於成本攤平。跑 AI 推論的 GPU（高速運算晶片）、向量資料庫、模型 API（讓程式呼叫 AI 的接口）每一項都所費不貲。若為每個客戶都獨立蓋一整棟，閒置資源與維運成本會迅速失控——就像為每位房客各蓋一棟透天，多數時間房子是空的，你卻得付十份水電。多租戶讓大家共享同一批資源，提升使用率、壓低單位成本，連功能升級與資安修補都只需做一次。

但天下沒有白吃的午餐。共用帶來低成本，也帶來一個不能破的底線：A 客戶絕對不能看到、用到、或被牽連到 B 客戶。下面這張表，幫你一眼看懂「共用一棟（邏輯隔離）」與「各蓋一棟（實體隔離）」的取捨。

多租戶的五個關鍵設計

把多租戶做對，其實就是把一條「請求的旅程」上的五道關卡都顧好。一個客戶的請求進來後，會依序經過：先確認你是誰（身分）→ 確保你只碰得到自己的資料（隔離）→ 看你還有沒有額度（配額）→ 決定用哪顆 AI 模型回答（路由）→ 最後記下這筆花了多少錢（計費）。任何一關漏掉，輕則算錯帳，重則資料外洩。

01

身分（你是誰）

用每租戶獨立的 API 金鑰辨識來源，套上對應權限。

02

隔離（只碰自己的）

資料與 RAG 檢索都鎖在自己的範圍，看不到別家。

03

配額（還有額度嗎）

速率與用量上限，擋住吵鬧鄰居把資源吃光。

04

路由（派哪顆模型）

依付費層級與任務難易，挑最划算的模型回答。

05

計費（記下花費）

逐筆記錄 token 與成本，回答「誰花了多少」。

關卡二：資料隔離模型

資料隔離是多租戶最不能出錯的環節。邏輯隔離（共用資料庫）讓所有租戶共用同一套資料庫與資料表，靠每一筆資料上多貼一個租戶標籤（tenant_id，等於每筆資料都蓋上「這是哪家公司的」印章）來區分歸屬。優點是資源利用率高、好擴充；缺點是所有查詢都必須強制帶上這個標籤過濾，一旦某段程式忘了加條件，就可能撈到別家的資料。實務上會用 row-level security（資料庫層級的「列權限」控管）、或在程式框架層自動注入租戶條件，從根本收斂這種「忘了加條件」的風險。

實體隔離則為每個租戶配置獨立的資料庫與儲存空間，甚至獨立運算資源。隔離性與合規舉證最強，適合金融、醫療等高敏感場景，但成本與維運負擔也最高。

在 LLM 場景中還有一個最容易被忽略、卻最容易出包的地方：向量資料庫與 RAG 索引的隔離。所謂 RAG（檢索增強生成，讓 AI 先查過你自己的資料、再根據查到的內容回答），會把企業文件切成小塊、轉成「向量」後存進向量資料庫。若不同租戶的資料混在同一個索引裡，AI 在檢索時極可能撈到別家公司的內容，再大方地寫進回答裡——這就是跨租戶外洩。正確做法是用 命名空間（namespace，可想成同一個大倉庫裡，每家公司有自己上鎖的專屬貨架） 把每個租戶的資料分開存放，並在每次檢索時強制只在「這家自己的貨架」上搜尋。

小心

跨租戶外洩是多租戶的頭號災難。最常見的破口有三個：① 某段查詢忘了帶 tenant_id，撈到別家資料；② RAG 檢索的 namespace 過濾失效，AI 引用了別家文件；③ 對話記憶或快取沒依租戶分隔，B 客戶看到 A 客戶的上下文。任何一個發生，都可能瞬間賠掉客戶信任與合約。

關卡一：身分與存取控制

每個租戶應持有獨立的 API 金鑰（等同這家公司專屬的門禁卡），平台據此辨識請求來自哪一家、套用對應的權限與配額。租戶內部再用 RBAC（角色型存取控制，依「職位」發不同權限的鑰匙） 劃分管理者、一般使用者、唯讀等角色，控制誰能管金鑰、誰能看哪些文件——就像大樓裡，總經理有萬能鑰匙，工讀生只能進茶水間。

關鍵心法是：讓「租戶身分」像一條安全帶，從頭繫到尾。從請求驗證、資料查詢、RAG 檢索、組裝給 AI 的提示，一路到記錄回應，每一步都帶著且驗證租戶標籤，並在給 AI 的系統指示中明確限定「只能使用本租戶的檢索結果」。安全帶只要中間鬆開一格，就等於沒繫。

關卡三：用量配額與速率限制

共用資源最大的營運風險，是吵鬧鄰居問題（noisy neighbor）：某個租戶突然狂打請求，把運算資源吃光，害得整棟大樓其他房客一起停水停電——回應變慢、甚至服務中斷。就像有人在公寓裡開了三十台冷氣，全棟跳電。對策是替每位房客裝上「電表與斷路器」。

速率限制（rate limiting，限制單位時間內能打幾次）控制每分鐘的請求數或 token 數，避免瞬間洪峰壓垮系統；token 配額則限制租戶在一個計費週期內可用的「總油量」，用完就降速或暫停。所謂 token（AI 計價與計算的最小單位，大約等於「字」的零件），輸入越長、輸出越多，就燒越多 token、花越多錢。建議同時設「軟上限」（接近時先告警、先限速）與「硬上限」（真的超標就阻斷），讓任何單一租戶都無法獨佔資源。

關卡四：模型路由與資源管理

不同租戶的需求與付費層級不同，可透過模型路由（依情況自動派工給不同的 AI 模型）把每個請求導向最划算的模型：付費高階方案用能力強的大模型，基礎方案用更便宜、更快的小模型；甚至同一個租戶內，也能依任務難易動態選——簡單的分類、改錯字用小模型，複雜的推理、寫長文才升級到大模型。這就像派工：搬個小件叫機車快遞，整櫃貨才出動貨櫃車，不會用貨櫃車去送一封信。

省成本的另兩大利器是快取與批次。快取（cache）是把重複或相似的問題「上次的答案」存起來重用，省下重算的錢；但快取務必以租戶為界，絕不能讓不同租戶共用同一份內容快取，否則又是外洩破口。批次（batching）則是把多筆請求湊在一起一次處理、提高 GPU 使用率，適合離線或不急著要結果的工作。下圖示意這些手段加總後，能把模型相關成本壓到什麼程度。

關卡五：成本歸屬與計量

多租戶要能長久經營，必須隨時答得出「每個租戶花了多少、該收多少」。核心是計量（metering，等於替每間辦公室裝獨立電表）：依每次呼叫的輸入與輸出 token 數、呼叫次數、用了哪一級模型，逐筆記下用量並貼上租戶標籤。沒有電表，就只能收統一管理費，最後不是虧錢、就是吵架。

這需要完整的可觀測性（observability，把系統內部運作攤在儀表板上看得見）支撐：每一次模型呼叫都留下含租戶、模型、token、延遲、成本的紀錄，匯總成各租戶的用量儀表板。這份資料一兼二用——既是帳務（billing）的依據，也能即時抓出異常用量與成本暴衝。最常見的成本分佈，是少數重度租戶吃掉大部分資源：

典型平台的成本集中度（少數租戶吃掉多數資源）
80%

常見約 20% 的重度租戶，吃掉約 80% 的運算成本。

若沒有逐租戶計量，你會用「平均」去訂價——結果是補貼了重度用戶、卻嚇跑了輕度用戶。有了電表，就能設計分級訂價、對症優化高成本租戶，把共用成本公平地分攤回去。

給中小企業／SaaS 的實務檢查清單

對資源有限的團隊，不必一開始就追求最完美的架構，但有幾件事必須從第一天就做對，否則日後幾乎補不回來：

建議

① 隔離：初期走邏輯隔離（共用大樓）省成本，但 tenant_id 過濾要從第一天就嚴格、自動化；向量庫一上線就用 namespace 隔離。
② 配額：速率限制與用量上限在「上線前」就要具備，先擋住吵鬧鄰居，再談優化。
③ 計量：可觀測性與逐租戶計量越早建立越好，否則無從訂價、也控不住成本。
④ 升級路徑：等到接到高合規或大型客戶時，再為其開「實體隔離／專屬部署」的升級選項，不必一開始就為所有人蓋透天。

給老闆的一句話

多租戶 LLM 系統的本質，是在「共享資源以攤平成本」與「嚴格隔離以保障信任」之間取得平衡。把隔離、配額、計量這三件事在架構初期就做對，平台才能在不犧牲任何一位客戶信任的前提下，安全地把一套系統規模化賣給一千家客戶。隔離守住信任、配額守住穩定、計量守住獲利——這三道防線，撐起整門生意。
